# Quality Assurance Plan — Sentinel Gateway

## 1. Purpose

This plan defines the quality assurance approach per ASPICE SUP.1, ensuring all work products and processes comply with defined criteria and non-conformances are resolved.

## 2. QA Organization

| Role | Responsibility |
|------|---------------|
| QA Lead | Defines criteria, conducts reviews, reports non-conformances |
| Developer Agent | Produces work products, addresses findings |
| Review Agent | Independent technical review |
| Product Owner | Approves deviations, resolves escalations |

**Independence**: QA reviews are conducted by agents/reviewers not involved in producing the work product under review.

## 3. Quality Criteria

### 3.1 Work Product Quality

| Criterion | Metric | Target |
|-----------|--------|--------|
| Document completeness | All YAML metadata fields present | 100% |
| Requirement quality | Testable acceptance criteria | 100% of requirements |
| Traceability coverage | Requirements with forward/backward links | 100% |
| Architecture consistency | Interfaces match implementation | Zero mismatches |
| Code compliance | MISRA C:2012 Required rule violations | Zero |
| Test coverage | Statement coverage | ≥95% |
| Test effectiveness | Requirement-to-test mapping | 100% |
| Diagram accuracy | Diagrams match text descriptions | Zero conflicts |

### 3.2 Process Quality

| Criterion | Metric | Target |
|-----------|--------|--------|
| Phase gate compliance | Gates passed before advancing | 100% |
| Review completion | Documents reviewed before approval | 100% |
| Change control | Changes tracked with rationale | 100% |
| Defect resolution | Open critical defects at release | Zero |

## 4. QA Activities by Phase

| Phase | QA Activities |
|-------|-------------|
| Phase 1: Requirements | Verify completeness, testability, traceability of STKH/SRS/SWRS |
| Phase 2: Architecture | Verify requirement allocation, interface completeness, ADR quality |
| Phase 3: Container Setup | Verify build reproducibility, SIL environment correctness |
| Phase 4: Implementation | MISRA compliance, code review, Doxygen completeness |
| Phase 5: Unit Test | Coverage verification, test quality review |
| Phase 6: Integration Test | SIL test execution, fault injection coverage |
| Phase 7: System Qualification | Full V-model verification, traceability audit |
| Phase 8: Review | Final compliance check, document package verification |

## 5. Review Process

### 5.1 Review Types

| Type | Scope | Participants | Formality |
|------|-------|-------------|-----------|
| Self-review | Individual work products | Author | Informal |
| Peer review | Technical correctness | Author + reviewer | Semi-formal |
| QA audit | Process compliance | QA Lead | Formal |
| Phase gate | Phase completion criteria | All roles | Formal |

### 5.2 Review Checklist Template

For each work product review:
- [ ] YAML front matter complete and correct
- [ ] Document ID unique and follows naming convention
- [ ] Version, date, status fields current
- [ ] All cross-references valid (no broken links)
- [ ] Requirement IDs unique and sequential
- [ ] Traceability links verified (forward and backward)
- [ ] Diagrams match text descriptions
- [ ] Acceptance criteria are measurable
- [ ] Safety classifications assigned where applicable
- [ ] No TODO/FIXME/placeholder content remaining

## 6. Non-Conformance Management

### 6.1 Severity Levels

| Level | Definition | Response Time |
|-------|-----------|---------------|
| Critical | Safety-relevant defect, wrong requirement, broken traceability | Immediate (block phase) |
| Major | Missing content, inconsistency, unclear specification | Before next phase gate |
| Minor | Formatting, typos, style issues | Before release |

### 6.2 Resolution Process

1. Identify non-conformance during review/audit
2. Log in `docs/reviews/non_conformance_log.md` with severity
3. Assign to responsible party
4. Fix and re-review
5. Verify fix and close
6. Escalate if unresolved within response time

## 7. Verification of AI-Generated Content

Since work products are generated by AI agents, additional QA checks apply:

| Check | Purpose |
|-------|---------|
| Hallucination detection | Verify requirements trace to real stakeholder needs |
| Consistency cross-check | Compare values across documents (timing, port numbers, etc.) |
| Completeness audit | Verify no requirements dropped during decomposition |
| Technical accuracy | Verify architectural decisions are feasible |
| Standard compliance | Verify ASPICE work product structure |

## 8. Records

All QA records maintained in:
- `docs/reviews/` — Review records
- `docs/reviews/non_conformance_log.md` — Non-conformance tracker
- `docs/project/project_plan.md` — Phase gate status
